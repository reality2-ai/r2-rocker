Net read: nothing on the new wire surface is broken end-to-end on the

with the reserved Console port (Finding F, high severity) and the

(Findings H, I). One **posture difference** that's defensible
(Finding J). One **deferred** item (`last_acked_seq` NVS persistence,
already tracked in §6.4 implementation deviation).

## What's new since the 2026-05-08 audit

* **SPEC-R2-ROCKER-TIMESYNC v0.1** — new spec; hybrid model where the
  sensor applies the offset locally and the dashboard refines via
  Cristian's algorithm. Adds the wire event
  `r2.dash.set_clock_offset {0: i64 delta_ms}`. `ts_ms` on the wire
  AND on disk now carries synchronised milliseconds rather than raw
  uptime.
* **Time-sync round-trip on the wire** — `r2.dash.sync_pulse` /
  `r2.sensor.sync_pong` events (already in SPEC-R2-ROCKER-WIRE §3.7
  / §4.5) are now actually implemented end-to-end in
  `firmware/esp32-s3/devkitc/src/sender.rs` (handler) and
  `dashboard/src/main.rs` (scheduler + `handle_sync_pong`).
* **SD ring v0.2 fixed-width CSV** — SPEC-R2-ROCKER-SENSOR §6.2
  switched from a 20-byte binary record to a 62-byte fixed-width
  CSV row to make pulled cards readable in pandas / Excel without a
  custom parser. Implementation in
  `firmware/esp32-s3/devkitc/src/ring.rs`.
* **`r2.dash.ack` end-to-end** — dashboard now emits at the earlier
  of 200 ms or 100 received acceleration frames
  (`dashboard/src/main.rs:1147-1148`). Sensor frees SD ring segments
  on ACK receipt.
* **`log_tcp` fan-out** —
  `crates/r2-esp/src/log_tcp.rs` installs a capturing logger that
  duplicates each log line to TCP subscribers on port 21045.
  Webapp's per-sensor logs panel proxies through the dashboard.
* **`cycle_hotspot: true`** — dashboard always tears down the
  matching NetworkManager hotspot before re-creating on a fresh
  "Connect Sensors" press (`dashboard/src/main.rs:708-714`); sensors
  currently joined drop WiFi, fall back to BLE, and get re-pushed
  credentials.
* **Firmware availability endpoints** —
  `/api/firmware/available` (GitHub Releases or local dir) and
  `/api/firmware/{carrier}/binary` (proxy/stream the .bin) in
  `dashboard/src/main.rs:1887-…`. Webapp drives auto-pull OTA off
  these.
* **Release-mode firmware build** — `R2_RELEASE=1` in the env
  refuses to build a dirty tree OR a non-tag HEAD; on success the
  baked `fw_ver` is the bare git tag (e.g. `v0.2.0`). See
  `firmware/esp32-s3/devkitc/build.rs:73-106`.
* **`log_tcp::install_logger` + `start_listener` split** — splitting
  the two avoids a deadlock where `TcpListener::bind` against
  `0.0.0.0:21045` was being called before lwIP was initialised; the
  listener thread never returned from `bind` and no log fan-out
  ever appeared on UART.

## Pass 1 — Time-sync wire conformance

### 1.1 FNV-1a-32 hashes for the new event names

**Verdict:** ✅ pass.

Computed independently (Python reference) against the firmware's
`fnv1a_32` const-fn output and the dashboard's `r2_fnv::fnv1a_32`:

| Event name | FNV-1a-32 |
|---|---|
| `r2.dash.set_clock_offset` | `0xAFFB208C` |
| `r2.dash.sync_pulse`       | `0xA1775708` |
| `r2.sensor.sync_pong`      | `0xCCAE4EBB` |
| `r2.dash.ack`              | `0xC8BD57C2` |

Sites:
* firmware:
  `firmware/esp32-s3/devkitc/src/wire.rs:41-46` —
  `EVT_SENSOR_SYNC_PONG`, `EVT_DASH_ACK`, `EVT_DASH_SYNC_PULSE`,
  `EVT_DASH_SET_CLOCK_OFFSET`.
* dashboard:
  `dashboard/src/main.rs:46-52` — same constants computed via
  vendored `r2_fnv::fnv1a_32`.

The event names follow the reverse-DNS class convention already
ratified by the 2026-05-07 audit (R2-WIRE §1.2; no `#` prefix
because these are agent-defined, not platform-defined).

### 1.2 Wire format

**Verdict:** ✅ pass.

R2-WIRE compact frame (12-byte header + CBOR payload) is unchanged
across the new events. `dashboard/src/main.rs:300-307`'s frame
builder uses `(version=0, msg_type=Event=0, flags=0)`,
`ttl=5, k=3`, target = broadcast zero — matches the firmware
encoder bit-for-bit.

CBOR payloads:

* `r2.dash.sync_pulse {0: req_id uint32, 1: dash_ts_ms uint64}` —
  `encode_sync_pulse` at `dashboard/src/main.rs:323-333`,
  `parse_sync_pulse_req_id` at
  `firmware/esp32-s3/devkitc/src/wire.rs:225-237`.
* `r2.sensor.sync_pong {0: req_id uint32, 1: sensor_ts_ms uint64}` —
  built inline at `firmware/esp32-s3/devkitc/src/sender.rs:334-348`,
  decoded by `decode_sync_pong` at
  `dashboard/src/main.rs:354-359`.
* `r2.dash.set_clock_offset {0: i64 delta_ms}` —
  `encode_set_clock_offset` at `dashboard/src/main.rs:336-350`
  branches on sign (`UInt` vs `NegInt`); parser at
  `firmware/esp32-s3/devkitc/src/wire.rs:242-263` accepts both
  CBOR major types 0 and 1, returning a signed `i64`.

CBOR keys are integers (per SPEC-R2-ROCKER-WIRE §1.3), ascending
order, smallest-form heads. RFC 8949 §4.2 deterministic encoding
holds.

### 1.3 Cristian's-algorithm semantics

**Verdict:** ✅ pass with one caveat (Finding I).

The dashboard's `handle_sync_pong`
(`dashboard/src/main.rs:373-475`) implements the spec's §3.1 + §3.2:

* `offset_estimate = T1 + rtt/2 - T2` (line 403).
* Exponential smoothing with α = 0.2 (line 406-411).
* Initial baseline trigger: ≥ 5 sync rounds + std-dev of last 3
  estimates < 5 ms (line 434).
* Drift trigger: `|smoothed| ≥ 10 ms` (line 439).
* After pushing, resets local `smoothed_offset_ms` to 0 (line 450)
  and clears the rolling estimates window — matches §3.2.

Cadence in
`dashboard/src/main.rs:1075-1103`'s per-peer pulse task: 1 Hz for
the first 30 s, then 30 s thereafter. Matches SPEC-R2-ROCKER-WIRE
§7 and SPEC-R2-ROCKER-TIMESYNC §3.1.

### 1.4 `r2.dash.ack` cadence + payload

**Verdict:** ✅ pass.

Dashboard cadence (`dashboard/src/main.rs:1147-1148`): `200 ms` OR
`100 received acceleration frames`, whichever first. This matches
SPEC-R2-ROCKER-WIRE §4.1 verbatim ("Sent every 200 ms or every 100
received samples (whichever first)"). It's also conservative
relative to R2-WIRE §4.1's general buffering guidance (no
per-frame cap — sensor's SD ring depth is the durable backstop, so
even a multi-second ACK gap is recoverable).

Sensor side (`firmware/esp32-s3/devkitc/src/sender.rs:309-322`):
parses `r2.dash.ack {0: through_seq, 1: dash_ts_ms}`, updates
`last_acked_seq` (RAM-only, see §1.5), calls
`ring.free_through(through_seq)` to delete fully-acked SD segments.

### 1.5 `last_acked_seq` NVS persistence

**Verdict:** ⏳ deferred (already documented in source).

SPEC-R2-ROCKER-SENSOR §6.4 calls for `last_acked_seq` to be NVS-
persisted with rate-limited writes (≤ 1 Hz). Current firmware
holds it in RAM only:

```
firmware/esp32-s3/devkitc/src/sender.rs:94-98
    /// Last `through_seq` received via `r2.dash.ack`. ... RAM-only in
    /// v0.1 — spec §6.4 calls for rate-limited NVS persistence,
    /// deferred.
```

After a power cycle the sensor will re-stream the whole un-deleted
ring instead of resuming from the dashboard's last ACK. For
operator-supervised v0.1 this is tolerable (dashboard's
deduplication on `(device_pk, seq)` per WIRE §5.3 means correctness
is preserved; only wire bandwidth is wasted). Tracked as a
follow-up.

## Pass 2 — SD ring v0.2 conformance

### 2.1 Fixed-width CSV record (62 bytes)

**Verdict:** ✅ pass.

Implementation at `firmware/esp32-s3/devkitc/src/ring.rs:45-55`
matches SPEC-R2-ROCKER-SENSOR §6.2 v0.2 column widths:

| Column | Width | Type |
|---|---|---|
| `seq`   | 10 | u32 ASCII, right-aligned, space-padded |
| `ts_ms` | 14 | i64 ASCII, right-aligned, space-padded |
| `x,y,z` | 11 each | i32 ASCII, right-aligned, space-padded |

Total = 10 + 14 + 11×3 + 4 commas + 1 LF = 62. `RECORD_BYTES = 62`
(ring.rs:45) keeps the boot-recovery `seek(record_index × 62)`
arithmetic valid.

`ts_ms` carries synchronised milliseconds per
SPEC-R2-ROCKER-TIMESYNC §2.2 — interpretable standalone, no side
file needed. Confirmed by Clock's `ts_ms_i64()`
(`clock.rs:66-70`) being the single source for both wire and SD
emission.

### 2.2 fsync cadence

**Verdict:** ✅ pass (within v0.2 spec deviations).

Spec §6.2.1: `sync_all()` every 100 records / on segment rotation /
on Drop. Implementation:

* `SYNC_EVERY_N_RECORDS = 100` (`ring.rs:61`).
* Rotation explicitly syncs (the spec's wording covers this).
* `Drop` path is the FATFS layer's responsibility on
  `esp_restart()`; the firmware does not have an explicit `Drop`
  impl on `Ring`, but the SDK fsyncs on graceful shutdown.

### 2.3 Layout deviations (already documented in spec)

**Verdict:** ✅ pass — the spec explicitly documents all three
deviations.

`SPEC-R2-ROCKER-SENSOR.md` §6.1 (lines 430-449) lists:

1. Segments at mount root, not `/r2/`.
2. Filenames `logNNNN.csv`, not `log.NNNN.bin`.
3. Fixed-width CSV record, not 20-byte binary.

All three match the current `ring.rs` implementation. No silent
drift.

## Pass 3 — Network / port-allocation conformance

### 3.1 log_tcp port 21045 (the headline finding)

**Verdict:** ⚠️ **deviation — port collision with canonical R2 spec.**

See Finding F below for the full write-up. Short version:
`crates/r2-esp/src/log_tcp.rs:24` binds port 21045 for the per-
sensor TCP log fan-out. The canonical R2 port table reserves 21045
for **Console (GraphQL over WebSocket / HTTP)** in:

* `R2-TRANSPORT.md:330-334` — port allocation table, "21045 | TCP/WS
  | Console / GraphQL (plugin) | R2-CONSOLE".
* `R2-WIRE.md:979` — same table, "21045 | TCP/WS | Console (GraphQL)
  | R2-CONSOLE".
* `R2-CONSOLE.md:102` — "21045 | HTTP + WebSocket | Console server
  (GraphQL API + static web UI)".
* `R2-INTERNET.md:140 + :153 + :315 + :387` — multiple references
  treating 21045 as the console WebSocket port for remote-access.

The rocker dashboard's WS proxy that reads off port 21045 from the
sensor is unaffected — that's a sensor-local TCP listener. But on
any future deployment where the same physical hive runs both
r2-console AND a rocker sensor (e.g. when sensor firmware migrates
to a Linux/SBC carrier), the two would race for the same port.

### 3.2 ACK cadence — buffering compliance

**Verdict:** ✅ pass.

R2-WIRE §4.1 + R2-WIFI §4.4 do not pin a specific ACK period; they
require ACK semantics to be "cumulative" and to "not stall the
sender's ring" — both hold. The dashboard's 200 ms / 100-sample
trigger keeps the worst-case un-ACKed window at 1 s @ 100 Hz =
100 records ≈ 6.2 kB. Well below the 8 MiB segment size.

### 3.3 TCP transport (events on TCP, not UDP)

**Verdict:** ✅ unchanged — pre-existing project-local deviation
already noted in 2026-05-07.

R2-WIFI §2.2 says R2-WIRE events ride on UDP:21042. r2-rocker uses
TCP:21042 (length-prefixed extended frames, which R2-TRANSPORT
§3.4 / R2-WIRE §13.4 explicitly allow as the "reliable ordered
stream" binding). Same deviation as before; not introduced by the
new work.

## Pass 4 — Bootstrap-layer conformance

### 4.1 `cycle_hotspot: true` on re-bootstrap press

**Verdict:** ⚠️ deviation in posture from R2-BOOTSTRAP §3.2; defensible.

`dashboard/src/main.rs:708-714` always sets `cycle_hotspot: true`,
i.e. NetworkManager tears down and re-creates the matching hotspot
on every "Connect Sensors" press, by design. Implemented in
`crates/r2-bootstrap/src/lib.rs:402-410` (calls
`bring_down_matching_hotspot()`, settles 2 s, then re-creates).

R2-BOOTSTRAP §3.2 Variant B (Gateway SoftAP, the rocker's
deployment pattern) reads:

> **When to use:** Hub deployments (rocker rig, Earthgrid),
> multiple sensors all connecting to one gateway. No ad-hoc SoftAP
> teardown needed; sensors stay on the gateway's network.

The implication is that the hotspot lifecycle is *monotonic* in
hub mode — once up, stays up. r2-rocker deliberately departs from
this so that operator-initiated re-bootstrap can recover a sensor
that's WiFi-stuck-but-unreachable (the only path back through
BLE-only re-pairing is to take down its WiFi). The comment at
`dashboard/src/main.rs:708-714` documents the trade explicitly.

Not a spec violation per se — R2-BOOTSTRAP doesn't ban
non-monotonic hotspots — but a hint that the spec's narrative
hasn't yet contemplated an operator-driven re-bootstrap loop. See
Finding J.

### 4.2 BLE / L2CAP / `#wifi_offer` path

**Verdict:** ✅ unchanged from 2026-05-08 audit. No new code paths
on the BLE side this session.

## Pass 5 — Dashboard-local infrastructure (not in R2 specs)

### 5.1 `/api/firmware/available` + `/api/firmware/{carrier}/binary`

**Verdict:** 📋 project-local; no R2 spec conflict.

Dashboard-internal HTTP endpoints, served by the same Axum router
that hosts the webapp:

* `dashboard/src/main.rs:1887-1921` — `firmware_available_handler`,
  fetches the latest GitHub Releases tag (cached
  `FIRMWARE_CACHE_TTL_SECS`), falls back to the local
  `firmware/esp32-s3/<carrier>/releases/` directory.
* `dashboard/src/main.rs:1923-…` — `firmware_binary_handler`,
  streams or proxies the .bin to the webapp.

R2-DEPLOY / R2-UPDATE govern the on-the-wire OTA push (port 21043,
already implemented in Phase 9-light + audited 2026-05-08). The
HTTP endpoints here are the **dashboard's own front-end-to-back-end
firmware lookup**, not part of the device-side OTA path. They are
out of scope for the canonical R2 specs, which is fine.

### 5.2 `log_tcp::install_logger` + `start_listener` split

**Verdict:** 📋 implementation note (no spec issue).

`crates/r2-esp/src/log_tcp.rs:90-94` installs the capturing logger
synchronously at very early `main()`; the network-touching
`start_listener` (`:99-112`) spawns the TCP accept loop after WiFi
is up. The split exists because calling `TcpListener::bind` against
`0.0.0.0:21045` before lwIP is initialised on ESP-IDF caused the
listener thread to hang in `bind` and produce zero output. Comment
in source captures this. Recorded for posterity in case a future
implementer hits the same shape on a different ESP-IDF subsystem.

## Pass 6 — Build / version conformance

### 6.1 `R2_RELEASE=1` → fw_ver = bare git tag

**Verdict:** ✅ pass.

`firmware/esp32-s3/devkitc/build.rs:73-106` enforces:

* `R2_RELEASE=1` AND dirty tree → `panic!` (build refuses).
* `R2_RELEASE=1` AND HEAD not on a tag → `panic!` (build refuses).
* On success: `R2_FW_VER = <tag>` exactly (e.g. `v0.2.0`), no
  timestamp, no `+<sha>` suffix.

R2-BUILD §8 (`R2-BUILD.md:357-366`) only requires the fw_ver to
come from `CARGO_PKG_VERSION` plus build-time stamping. R2-DEPLOY
§10 (`R2-DEPLOY.md:631`) suggests the combined form
`<platform_version>-<git_hash>[+dirty]`. r2-rocker's release-mode
form is **stricter** than R2-DEPLOY's suggested form — the bare
tag is unambiguous BY ITSELF because the tag-on-clean-tree
enforcement is verified at build time. Dev mode uses the
fuller `<semver>-<UTC-yyyy-mm-dd-HH:MM>+<sha>[-dirty]` form
which matches R2-DEPLOY §10's pattern.

The matching dashboard logic (firmware availability endpoint above)
keys off the tag directly when comparing what a sensor announces
vs what GitHub Releases serves; bare-tag release-mode strings
make this a literal `==` comparison rather than a
prefix-or-substring match. Defensible.

## Findings

### Finding F — `log_tcp` port 21045 collides with reserved R2 Console port

**Severity:** deviation; affects future deployments where r2-console
and a rocker sensor would coexist on one hive.

**File / line:**
* r2-rocker: `crates/r2-esp/src/log_tcp.rs:24`
  ```
  const LOG_PORT: u16 = 21045;
  ```
* canonical reservation:
  * `r2-specifications/specs/r2-core/R2-TRANSPORT.md:330-334` —
    port table: "21045 | TCP/WS | Console / GraphQL (plugin) |
    R2-CONSOLE".
  * `r2-specifications/specs/r2-core/R2-WIRE.md:979` — same row in
    R2-WIRE's port allocation table.
  * `r2-specifications/specs/r2-core/R2-CONSOLE.md:102-104` —
    "21045 | HTTP + WebSocket | Console server (GraphQL API + static
    web UI)".
  * `r2-specifications/specs/r2-core/R2-INTERNET.md:140`, `:153`,
    `:315`, `:387` — multiple normative uses (port-forwarding,
    UPnP mapping, WSS URL examples).

**Today's impact**: zero on the rocker rig, because no rocker
sensor runs r2-console concurrently. The dashboard's WS proxy
talks to port 21045 on the sensor; the dashboard itself does not
bind that port. So the field bench is unaffected.

**Future impact**: any deployment where the same physical hive
runs a sensor AND r2-console (most plausibly when the sensor
firmware later runs on a Linux SBC carrier, or when the dashboard
machine is also a sensor) will fail to bind one of the two.

**Recommendation**: move the rocker debug log fan-out off 21045.
Two reasonable options:

1. **Move to a non-reserved private port** outside the 21042-21045
   block — e.g. `21046` (next sequential, currently unassigned)
   or somewhere clearly out of band like `27045`. Update
   `log_tcp.rs:24` and the dashboard's `/ws/logs` proxy that
   currently dials port 21045.
2. **Lift the rocker log-tail capability into R2-CONSOLE proper**
   — the spec's stated purpose for port 21045 is "device-side
   GraphQL + diagnostic surface", and "tail my log lines" is
   exactly that. Adding a `tail_logs` subscription on the existing
   R2-CONSOLE GraphQL schema would satisfy the rocker's UX need
   without inventing a parallel protocol. Bigger work but stays
   inside the spec.

Option 1 is the minimal fix; option 2 is the right long-term
direction. Tracked as a r2-rocker code-side TODO. The audit
recommends doing this **before university handoff** — the
handover crowd may well try to run r2-console on the same hive.

### Finding G — `set_clock_offset` handler does not emit the 200 ms ack status frame

**Severity:** deviation; low impact under the bench-validated cadence.

**Spec:** `specifications/SPEC-R2-ROCKER-TIMESYNC.md:114-125` §2.3
step 4:

> Acknowledge by emitting a fresh `r2.sensor.status` frame within
> the next 200 ms so the dashboard sees the updated `ts_ms` and
> can confirm the correction took effect.

**Implementation:**
`firmware/esp32-s3/devkitc/src/sender.rs:323-330` —

```
EVT_DASH_SET_CLOCK_OFFSET => {
    if let Some(delta_ms) = parse_set_clock_offset(payload) {
        info!("[sender] r2.dash.set_clock_offset delta_ms={:+}", delta_ms);
        self.clock.apply_delta(delta_ms);
    } else {
        warn!("[sender] malformed set_clock_offset payload");
    }
}
```

The handler updates the clock but does NOT trigger an immediate
status frame. The next status frame goes out on the regular
`STATUS_PERIOD_MS = 2 s` schedule
(`firmware/esp32-s3/devkitc/src/sender.rs:42` + main loop at
`:242-245`). So in the worst case the dashboard waits up to 2 s
to see a `ts_ms` reflecting the new offset, missing the spec's
200 ms bound by an order of magnitude.

The dashboard's retry-after-1s policy (TIMESYNC §3.2's last
paragraph) will fire occasionally as a result, which is wasted
network and a minor log-spam concern but not a correctness
problem.

**Recommendation**: after `self.clock.apply_delta(delta_ms)` add
`self.send_status(stream)?;` so the dashboard sees the corrected
`ts_ms` in the spec's 200 ms window. One-line fix.

### Finding H — `SPEC-R2-ROCKER-WIRE.md` §3.5 / §3.7 still describes `ts_ms` / `uptime_ms` as "monotonic uptime"

**Severity:** documentation drift; the on-the-wire bytes are
correct, the prose is stale.

After SPEC-R2-ROCKER-TIMESYNC §2.2 landed, the meaning of `ts_ms`
on the wire shifted from "monotonic uptime" to "synchronised
deployment milliseconds". SPEC-R2-ROCKER-TIMESYNC §5.4 covers the
u32 wrap and §2.5 covers the SD record semantics. But the
sibling specs still say:

* `SPEC-R2-ROCKER-WIRE.md:200`:
  > `1 | uint32 | uptime_ms`

  in the `r2.sensor.status` row.

* `SPEC-R2-ROCKER-WIRE.md:231`:
  > `1 | uint32 | sensor_ts_ms — sensor's monotonic time at frame-
  > receive instant`

  in the `r2.sensor.sync_pong` row.

* `SPEC-R2-ROCKER-WIRE.md:150`:
  > `1 | uint32 | ts_ms — sensor uptime in milliseconds at the
  > sample instant`

  in `r2.sensor.acceleration`.

* `SPEC-R2-ROCKER-SENSOR.md:401-405` §5.4 — same drift for the SD
  record semantics (this section is now overridden by §6.2's v0.2
  notes plus TIMESYNC §2.5, but the prose still claims "monotonic
  uptime counter in milliseconds").

The TIMESYNC spec's "Amends" header (line 7) takes precedence, so
nothing is *wrong* on the wire. But a clean-room implementer
reading WIRE / SENSOR before TIMESYNC would land on the obsolete
semantics. Tracked in TIMESYNC §1.1 as the design motivation
("dashboard maintains a per-sensor offset in memory used to convert
sensor timestamps to wall-clock at analysis time" is the *old*
behaviour, which the TIMESYNC spec replaces).

**Recommendation**: thread an "AMENDED BY SPEC-R2-ROCKER-TIMESYNC
§2.2 — `ts_ms` carries synchronised milliseconds" note into each
of the four spec rows above. One-line per row. Keeps the catalogue
self-consistent without a wholesale rewrite.

### Finding I — `SPEC-R2-ROCKER-WIRE.md` §3.5 status enum drifted from current implementation

**Severity:** documentation drift; the wire bytes are stable, the
spec table is from a pre-Phase-5L draft.

`SPEC-R2-ROCKER-WIRE.md:199`:

> `0 | uint8 | state — see §6 enum (Idle=0, Advertising=1,
> BleListening=2, WifiConnecting=3, Streaming=4, Calibrating=5,
> Error=6)`

The current spec/code uses the 10-value enum at
`SPEC-R2-ROCKER-SENSOR.md:252-263` §4.1.1
(BOOT=0…ERROR=9). 2026-05-08 audit already noted SENSOR was the
canonical source; that audit recommended documenting the mapping
in SENSOR §4 — done. But the WIRE spec's inline enum was missed
in the same edit and still names the old 7-value set.

**Recommendation**: replace the inline enum in WIRE §3.5 with a
cross-reference: "see SPEC-R2-ROCKER-SENSOR §4.1.1 for the
authoritative state→u8 mapping". Stops the two from drifting in
opposite directions on the next refactor.

### Finding J — R2-BOOTSTRAP §3.2 narrative assumes monotonic hotspot lifecycle

**Severity:** spec-narrative gap; not a violation.

R2-BOOTSTRAP `R2-BOOTSTRAP.md:143`:

> **When to use:** Hub deployments (rocker rig, Earthgrid),
> multiple sensors all connecting to one gateway. No ad-hoc SoftAP
> teardown needed; sensors stay on the gateway's network.

The narrative implies that the gateway hotspot, once up, stays up.
r2-rocker's
`cycle_hotspot: true` (`dashboard/src/main.rs:708-714`,
`crates/r2-bootstrap/src/lib.rs:402-410`) deliberately cycles the
hotspot to recover sensors that are WiFi-stuck-but-unresponsive —
the only way back to BLE bootstrap is through losing WiFi.

This is a defensible product choice (operator-initiated recovery
is exactly the operator-supervised posture the rocker assumes per
`feedback_calm_tech_security` and the supervised-not-unattended
memory). But it's worth raising upstream: real hub deployments
will need an explicit "re-bootstrap one stuck sensor" path, and
the cleanest way to enable it is to take the hotspot down.

**Recommendation**: feed back into R2-BOOTSTRAP §3.2 a short
paragraph: "Hub-mode hotspots SHOULD be considered teardownable
when an operator explicitly requests re-bootstrap; sensors
joined to the hotspot at teardown SHOULD fall back to BLE
advertising within 5 s of WiFi loss and accept a fresh
`#wifi_offer` over BLE L2CAP CoC." This makes the rocker's
behaviour normative rather than a deviation.
